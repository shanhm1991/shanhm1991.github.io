Maven

Maven是Java应用中一种很常见的构建方式，这里记录下我们使用Maven来构建springboot应用的一次实践以及遇到的问题

示例

在项目实施过程中，一般会将一些通用的处理提取成公共的依赖。我们这里使用commons-framework来作为公共处理，并努力使其与具体的项目无关。另外为了统一项目中使用的依赖版本，定义commons-dependencies，在继承spring-boot-dependencies的基础上添加了一些自己要用的依赖版本。

具体到项目的某个微服务中，我们可以使用parent-model形式来组织自己的代码模块，通常各服务模块之间会相互调用，一般会单独使用一个模块来定义数据模型，方便提供给其他服务使用。这样只用在parent模块中统一引用commons-framework，并使用commons-dependencies来管理依赖版本

但这样还是觉得太麻烦了，导致jar包泛滥不好管理。所以我们使用一个sys-model来管理项目中所有服务的数据模型（根据实际情况，也可以将相关的服务进行分组来共用model），sys-model会依赖commons-framework，然后各个服务再依赖sys-model，它们统一使用commons-dependencies来管理依赖版本

commons-dependencies的作用是用来约束依赖版本，所以依赖全部定义在dependencyManagement中，并通过import继承了spring-boot-dependencies

在具体服务中，会使用commons-dependencies来管理依赖版本，并使用sys-model作为依赖，间接依赖于commons-framework

scope 作用域

  • compile

默认的scope，适用于所有阶段。最终会打包到artifact中，如果构建的是一个WAR类型artifact，那么compile引用的Jar会被集成到WAR文件中。

  • provided

适用于编译和测试阶段。这种在类似commons-framework这样的公用模块中比较常用，它不会打包到artifact中，也不会传递依赖关系。其含义是认为对应的依赖会由运行这个应用的JDK或者容器提供。

  • runtime

适用于运行和测试阶段。与compile相比，依赖的项目无需参与编译。比如Jdbc的驱动包，项目编译只需要JDK提供的JDBC接口，只有在执行测试或者运行时才需要具体的Jdbc驱动。

  • test

仅适用于测试阶段。不参与编译和打包，只在编译或运行测试代码的时候才使用，比如Junit。

  • system

类似于provided，区别在于你需要告诉Maven如何去找到这个依赖。所以当引用的依赖在Maven仓库中不存在时，可以使用，但是不推荐。如果希望依赖参与打包，需要特别指定，比如下面这样

  • import

只能在dependencyManagement中使用，用于从其它的pom文件中导入其所有依赖设置，比如commons-dependencies中所示。

plugin 插件

  • maven-compiler-plugin

compile是默认就存在的操作，可以通过插件指定一些配置，比如<extdirs>，表示可以额外从执行目录中获取依赖

  • maven-jar-plugin

jar也是默认的操作，通过插件主要配置下jar包中包含或排除哪些内容，这里将config目录放在jar包外，方便运行时修改配置文件，另外也是希望与以前的项目保持同样的结构

  • smart-doc-maven-plugin

smart-doc的插件，在编译阶段根据注释生成应用接口文档

  • spring-boot-maven-plugin

springboot提供的打包方式，将应用jar，以及所有依赖的jar合并起来打包，并做成一个可以直接java -jar xxx.jar运行的包

以前是使用maven-assembly-plugin将所有文件打成一个tgz，但是在替换环境jar包调试时（坏习惯，以前常用的操作），经常因为依赖的jar没有替换而导致问题，因为jar包只包含当前编译的代码。在使用springboot打包则没有这种问题，它会将依赖打包成一个整体，而且有对应的插件处理jar包加密问题

对应的打包描述文件，意思就是选取需要打包的文件并放到目标目录，这里将bin下面的install.sh挪到了根目录下

  • classfinal-maven-plugin

jar包加密，目的是防止反编译，仅针对spring-boot-maven-plugin打出来的jar。这里将执行阶段phase设置成none相当于关闭了，如果设置成package，那么在启动参数中需要添加 -javaagent:/{path-to-lib}/xxx.jar

  • maven-release-plugin/maven-scm-plugin

解决打包版本控制的一个插件，分为快照版本和发行版本，所以要配两个仓库。本地开发版本永远是snapshots快照，在发布一个releases之后，它会帮忙将代码一个tag并推送仓库，然后将本地升级到下一个快照

在jenkins中构建时，我们会添加一个参数来表示是否要发布版本

如果这样需要指定拉取的本地分支，因为jenkin拉取代码时会分离Head，这会导致插件push代码失败

附录

  • jar引用失败

本地存在依赖，但是构建时依然提示依赖获取失败，并从远程获取。遇到这种问题可以尝试删除_remote.repositories文件，其作用是用来标示资源的来源，可以帮助Maven确定资源是否来自远程仓库，以及是否需要从远程仓库获取更新

有时我们在离线环境上构建，也会修改maven的mirror地址改为本地文件
