JUC ThreadPoolExecutor

———— JDK 1.8
words: 5.4k    views:    time: 22min
Java SE
ThreadPool

Executor是java中任务执行的主要抽象，虽然它是一个简单的接口，但却为灵活而强大的异步任务执行框架提供了基础，它提供了一种标准的方法将任务的提交过程与执行过程解耦开来。

ExecutorService在Executor的基础上提供了对执行服务生命周期的管理，ThreadPoolExecutor作为具体实现，提供了对线程Thread以及任务Runnable的管理，其基本思路是利用阻塞队列对任务进行管理，调用者只负责提交任务到队列，然后自己管理的线程负责从队列中获取并执行任务。ThreadPoolExecutor最主要的职责就是负责在何时以怎样的方式创建线程，然后又在怎样的时机让线程结束，其主要目的就是希望在实现任务异步执行的同时能尽量复用线程，避免无节制的频繁创建造成对系统资源的开销

这里并不打算去整理线程池的使用方法或注意事项，而是主要梳理下实现思路，尽量去理解和体会作者各种设计的巧妙及其背后的意图，其中利用了大量的CAS无锁操作，并充分发挥了工作线程的主观能动性来解决问题

1. 属性

1.1. ctl 状态 & 线程计数

线程池定义了5个状态，即：

  • RUNNING： 运行状态，可接收处理新任务
  • SHUTDOW： 关闭状态，处理队列中剩余的任务，但不接受新任务
  • STOP： 停止状态，拒绝接受新任务，并且也不处理队列中的任务，同时中断正在处理中的任务
  • TIDYING： 整理状态，所有任务线程结束，整理释放资源，将执行terminated()方法
  • TERMINATED：结束状态，terminated()方法执行完成

由于状态转换都是单向的，所以可以将状态值定义成不同的数值，然后直接从数值大小关系上判断和比较状态

由于只有5个状态，而且只要保证大小关系确定即可。那么，可以将SHUTDOWN状态定义为0，然后将RUNNING状初始化为负数，这样就可以通过RUNNING状态值的加减来描述worker数的变化了，只要结果不超过0就不会影响状态判断。

于是，又想到了通过二进制位运算来表达，将Int的前3位用来表示线程池的状态，刚好首位取1为负数（java通过补码表示数值，可以参考之前笔记：java字符串. 关于原码、反码、补码），另外两位有4种排列组合用来表示其余的状态，剩下的29位则可以用来记录worker数，而且可以通过加减RUNNING状态值来记录worker数的变化，只要结果不超过0就不会影响状态的判断。

但是如何获取具体的状态，以及worker数呢？RUNNING状态值在设计时也有特别考虑，其将前三位全部置1，后29位置0，然后按位取反定义了一个CAPACITY用来辅助计算，这样便可以通过运算ctl & CAPACITY得到当前实际的worker数，而ctl & ~CAPACITY则可以得到当前的实际状态值，只需一个ctl变量便同时纪录了线程池的状态和worker数两个信息。

1.2. workers

内部使用Worker来封装执行线程，它也是基于AQS的一个实现，其在初始化时setState(-1)，在unlock中setState(0)，然后在lock中执行compareAndSetState(0, 1)，也就是说Worker利用AQS来实现了一个反向操作，即先有释放锁，然后才能获取锁。这样设计的目的是为了防止在关闭空闲线程时，误伤了正在执行任务的线程

至于具体的执行还是委托给Thread，其负责拉起真正的线程并执行任务，思路就是循环从任务队列中获取并执行任务

3. 接口

3.1. execute

execute作为提交任务执行的入口，并没有过多的逻辑，基本都是放在addWorker中以便复用，如果结合addWorker中的处理可以总结下线程池进行reject的两种场景，即线程已经关闭了，或者线程数达到max，并且任务队列已经满了。另外，这里可以总结下线程池创建工作线程的策略：

  1. 线程数 < core，且任务队列未满，则直接创建线程执行任务
  2. 线程数 >= core，且任务队列未满，则提交任务到队列
  3. 线程数 >= core，且任务队列已满，则直接创建线程执行任务
  4. 线程数 >= max，且任务队列已满，则reject

从任务被提交到任务被执行的过程中有很多步骤，但是如何保证在这些步骤之间能及时感知到线程池的状态变化呢？其实也没有什么好办法，就是每走一步就检测一次，比如从上面的流程图（进入addWorker的第一步也是状态检测）

3.1.1. addWorker

addWorker中负责创建 worker 线程，并进行启动。不过它有个创建前提，只有当线程计数成功加 1 时，才进行创建，并且通过循环 CAS 操作来应对可能出现的竞争问题，即同时多个线程提交任务。至于这样先计数，后创建的方式，则是为了确保在并发情况下实际创建的 woker 数不会超过限制。

3.1.2. runWorker

runWorker由具体的工作线程执行，工作线程是在addWorker中被提交线程拉起的，但是已经与提交线程没有关系了，它只负责从队列中循环获取和执行任务，只有当获取任务为空或者执行任务异常时才结束自己

3.1.2.1. getTask

getTask的返回值承担了是否结束自己的决定，并利用队列的获取操作来检测对于空闲线程中断的检测，具体的流程步骤基本都在代码注释中描述清楚了，这里也不再过多赘述

3.1.2.2. processWorkerExit

processWorkerExit就是在线程结束自己时，做一些检查和善后工作，比如如果自己是因为异常结束，或者自己结束后线程数小于最低限制，那么应该帮忙拉起一个新的worker来代替自己

3.2. shutdown

线程池作为执行服务，可以响应外部的关闭请求，其处理就是将状态置为SHUTDOWN，并尝试中断工作线程。对应到getTask中响应中断的处理，其并非让线程在检测到中断时就直接结束，而是重新检测线程池状态和任务队列，只有当任务为空时，才会结束自己。这样就保证了在shutdown，依然会多线程处理队列中的任务，直到处理完了再各自结束

3.3. shutdownNow

shutdownNow更直接一点，直接将状态置为STOP，还是对应getTask中的处理，工作线程在看到STOP时直接结束，至于队列中剩余的任务则直接抛给调用者

3.4. awaitTermination

shutdown和shutdownNow调用只是通知了线程池尽快结束，但是具体什么时候真正结束了则无从知晓。因此，线程池提供了一个同步等待的调用，即awaitTermination来让调用者知道自己完成了关闭。在上面的tryTerminate中，其在完成terminated之后会进行条件唤醒操作，以通知阻塞在awaitTermination上的调用者。

3.5. setMaximumPoolSize

线程池支持参数的实时设置，如果当前的线程数超过了要设置的max，则尝试结束一些空闲线程，具体看上面的实现都是类似的

3.6. setCorePoolSize

设置核心线程数也类似，如果设置的值小于原来的值，则尝试中断空闲线程，但是反过来则要结合任务队列中实际剩余的任务数来创建，因为如果队列中没有要处理的任务，那么创建线程也是没有意义的

3.7. setKeepAliveTime

设置空闲时间比较简单，其实就是设置从队列中获取任务的阻塞实际，如果比原来小了，就中断然后重新进入阻塞并计算超时

3.8. getActiveCount

另外，线程池提供了一些获取其当前状态信息的接口，比如当前执行的任务数，这样结合执行过程中的lock操作，就可以很容易的获取哪些线程正在执行任务。至于其它的获取等待的任务数，完成的任务数则比较简单，就不再赘述了。
